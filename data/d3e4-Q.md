The overflow check in FraxlendPairCore.sol#L471-L474 is pointless as the possibility of it happening defeats the capacity of the vault to store value, and were it to happen the contract is already effectively broken. It must be expected to be sufficient, or uint256 could be used, which was designed to be plenty sufficient. If `_interestEarned` overflows it is certainly valuable enough to be recorded, but this check, if overflow happens, effectively steals funds from the lenders.

`_deposit()` in FraxlendPairCore.sol#L559-L576 accepts arbitrary values of `_amount` and `_shares`, which opens the door to incorrect accounting. This has been done to allow for either to be specified and thence derive the other in `deposit()` (L583-L595) and `mint()` (L602-L614). A neater and safer solution might be to merge `_deposit()` and `deposit()` to take as a parameter only `_amount` and therein calculate `_shares`, and let `mint()` translate its `_shares` into `_amount` and with this call this new `deposit()`. The old solution always does one translation. This new one does one, or two (with `mint()` then `_shares` would be translated into `_amount` and then back again), but the upside is that we can reduce three functions to two and save some lines of code.
